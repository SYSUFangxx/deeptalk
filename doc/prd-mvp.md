# 夫妻深夜话题生成器 MVP PRD

## 1. 项目背景

很多夫妻在日常生活中，尤其是睡前相处时，常常会遇到“想交流，但不知道聊什么”的情况。普通闲聊容易重复、浅层，长期下来会影响新鲜感、理解感和亲密感。

本产品希望通过一个轻量、易访问的 Web 应用，为夫妻提供高质量、可直接开启对话的话题，帮助双方更自然地沟通、回忆、表达和理解彼此。

## 2. 项目目标

打造一个可在手机访问的 Web 应用，让夫妻能够随时获得适合夜聊的高质量话题，并快速开始交流。

MVP 阶段重点验证三件事：

1. 用户是否愿意使用“话题生成”来开启睡前交流。
2. 话题内容是否足够自然、有价值、适合夫妻场景。
3. 话题分类和换题机制是否足以支持持续使用。

## 3. 目标用户

目标用户为已婚夫妻。

用户特征：

1. 希望增进沟通和亲密感。
2. 希望在日常生活中获得轻量、低门槛的交流工具。
3. 更偏好手机端使用。
4. 不希望产品使用成本高、学习成本高。
5. 对隐私、内容边界、情绪安全较敏感。

本阶段不考虑情侣用户，因此话题风格、分类、边界控制都围绕婚姻关系展开。

## 4. 产品定位

这是一个面向夫妻的夜聊话题工具，不是社交平台，不是婚恋咨询平台，也不是复杂的关系分析工具。

MVP 的核心定位：

打开就能用，一键就能得到一个值得聊的话题。

## 5. 核心用户场景

1. 睡前两个人躺下后，不知道该聊什么。
2. 想认真聊聊彼此，但不想把气氛搞得太严肃。
3. 希望偶尔回顾感情、谈谈未来、聊聊婚后日常。
4. 某些题不想聊，希望快速跳过。

## 6. MVP 范围

MVP 只做最基础、最核心的能力，聚焦“生成话题”。

包含功能：

1. 随机生成话题
2. 按分类筛选话题
3. 话题卡片展示
4. 换一个
5. 移动端优先的 Web 体验

不包含功能：

1. 用户注册登录
2. 多端同步
3. 智能推荐算法
4. 自动生成 1000+ 新话题
5. 收藏功能
6. 本地存储
7. 双人答题模式
8. 对话记录与心得
9. 语音功能
10. 社区分享
11. 专家话题包
12. 关系健康分析

## 7. 功能需求

### 7.1 首页：抽卡生成话题

用户进入首页后，应看到一个抽卡式入口。首页不直接展示分类信息，核心动作是抽出今晚的话题。

功能要求：

1. 默认展示一张未翻开的“今晚签”卡片。
2. 提供“抽今晚的话题”按钮。
3. 抽卡后展示 1 条话题。
4. 抽出话题后，主按钮变为“再抽一张”。
5. 首页只保留一个弱化的分类入口。
6. 话题展示要简洁清晰，适合手机阅读。

首页核心操作：

1. 抽今晚的话题 / 再抽一张
2. 按分类筛选

### 7.2 分类筛选

用户可以按主题方向查看或筛选话题。

MVP 建议分类：

1. 共同回忆
2. 日常相处
3. 未来规划
4. 价值观交流
5. 兴趣与生活
6. 轻松脑洞

功能要求：

1. 可查看分类列表。
2. 点击分类后，只展示该分类下的话题。
3. 分类内仍可继续“换一个”。

### 7.3 话题卡片展示

每条话题应以卡片形式展示。

建议字段：

1. 话题正文
2. 分类
3. 标签
4. 可选引导语

展示原则：

1. 一眼能看懂。
2. 文案温和，不像问卷。
3. 不堆太多信息，避免压迫感。

### 7.4 分类内换题

用户进入某个分类后，可以在该分类内继续换题。

功能要求：

1. 支持在当前分类内换一个话题。
2. 支持在当前分类内跳过当前话题。
3. 已跳过的话题在短期内尽量不重复出现。

## 8. 内容需求

### 8.1 内容来源

第一版话题内容来源于现有 100 个种子话题，但不能直接全量上线，需要经过二次筛选。

### 8.2 内容筛选原则

适合进入夫妻版 MVP 的话题，应满足：

1. 适合婚姻关系场景
2. 开放式，可自然展开
3. 不明显引导争吵
4. 不高频触发痛点或羞耻感
5. 有情感价值或趣味性

### 8.3 暂缓进入 MVP 的内容类型

1. 前任相关
2. 强刺激冲突题
3. 原生家庭创伤深挖
4. 高压试探型问题
5. 极端负面假设题

### 8.4 内容标签

每条话题建议具备基础标签，便于后续扩展。

建议标签：

1. 分类
2. 风格：轻松 / 温暖 / 深入 / 探索
3. 深度：浅 / 中 / 深
4. 敏感度：低 / 中
5. 时间向：过去 / 现在 / 未来

## 9. 非功能需求

### 9.1 终端要求

1. 优先适配手机浏览器。
2. 兼容桌面浏览器。
3. 页面加载轻量、打开快。
4. 适合单手浏览和点击。

### 9.2 体验要求

1. 首屏尽快看到话题。
2. 交互简单，不超过 1-2 次点击即可开始使用。
3. 风格温和、亲密、不过度花哨。
4. 避免内容显得像测试题或心理审问。

### 9.3 数据与隐私

1. MVP 阶段尽量少收集用户数据。
2. 不强制登录。
3. 不保存敏感对话内容。

## 10. 技术方案约束

你提到希望“使用 GitHub 作为服务端”，结合当前 MVP，建议产品设计层面先按轻服务端思路规划。

当前推荐的实现方向是：

1. 前端使用 Web 应用形式。
2. 话题数据、分类配置等可先托管在 GitHub 仓库中。
3. 网站可优先走静态部署方案，便于手机访问。
4. MVP 阶段尽量减少复杂后端依赖。
5. 首版尽量不引入用户持久化状态。

这里的产品假设是：

GitHub 在 MVP 阶段主要承担代码托管、静态内容存储、部署载体的角色，而不是完整业务后端。

如果后面需要账号、同步、推荐、数据分析，再逐步接入正式后端服务。

## 11. 页面结构

MVP 建议 2 个页面或区域：

1. 首页
当前话题展示，支持抽卡和再抽一张。
2. 分类页
展示分类列表，按分类浏览和生成话题。

## 12. 核心流程

### 12.1 随机生成流程

1. 用户打开首页
2. 系统展示未翻开的抽卡卡片
3. 用户点击“抽今晚的话题”
4. 系统翻开卡片并展示 1 条话题
5. 用户选择再抽一张或按分类筛选
6. 系统继续展示下一条话题

### 12.2 分类使用流程

1. 用户进入分类页
2. 选择一个分类
3. 系统展示该分类话题
4. 用户继续换题或跳过

## 14. 成功指标

MVP 阶段建议重点关注以下指标：

1. 访问后开始使用率
2. 单次会话浏览话题数
3. 跳过率
4. 分类使用率

MVP 重点先看方向对不对，不追求复杂指标体系。

## 15. 里程碑建议

### 阶段一：内容收敛

1. 从原始 100 题中筛选适合夫妻 MVP 的首发题库。
2. 完成分类、标签、敏感度标注。
3. 形成首发内容集。

### 阶段二：基础产品实现

1. 完成首页、分类页。
2. 完成随机生成、跳过。
3. 完成移动端适配。

### 阶段三：上线验证

1. 部署可访问 Web 版本。
2. 进行真实使用测试。
3. 根据使用反馈调整题库。

## 16. 风险与注意事项

1. 内容风险：有些题目虽然有讨论价值，但不适合夫妻睡前场景，容易引发压力或争执。
2. 体验风险：如果页面复杂、操作多，用户很快会失去耐心。
3. 价值风险：如果题目像“测试题”而不是“聊天引子”，用户会觉得生硬。
4. 技术风险：如果 MVP 一开始就把 GitHub 当成完整后端，会限制后续扩展，所以当前更适合把它当成轻量托管和发布基础设施。

## 17. MVP 结论

这个 MVP 的本质，不是做一个庞大的情感平台，而是做一个可随手打开、能立即开启交流的夫妻夜聊工具。第一阶段最重要的是：

1. 题库质量足够好
2. 使用路径足够短
3. 手机上足够顺手
4. 内容体验足够自然，愿意继续使用
