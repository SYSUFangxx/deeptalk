# 夫妻深夜话题生成器 MVP 技术方案草案

## 1. 文档目标

为 MVP 提供一个尽量轻、易部署、适合 GitHub 托管的实现方案。

## 2. 技术目标

本期技术方案要满足：

1. 做成 Web 应用，手机浏览器可直接访问。
2. 前后端复杂度尽量低。
3. 尽量利用 GitHub 做代码托管和发布。
4. 首版不依赖传统后端服务。
5. 直接打开本地 HTML 也能运行，方便快速验证。

## 3. 总体方案

当前 MVP 采用零构建静态前端 + GitHub 托管数据的架构。

架构分层：

1. 前端应用
负责页面渲染、话题展示、分类筛选、换题和跳过。

2. 静态数据层
话题数据、分类配置、标签配置以普通 JavaScript 数据文件形式保存在仓库内。

3. 部署层
使用 GitHub Pages 部署 Web 应用，使其可直接通过手机浏览器访问。

## 4. 为什么这样设计

优点：

1. 开发快，适合 MVP。
2. 不需要维护数据库和服务端接口。
3. 不需要安装依赖和构建步骤。
4. GitHub 直接承担代码管理和发布角色。
5. 后续可逐步升级，而不是推倒重来。

限制：

1. 数据不会跨设备同步。
2. 无法做真正的服务端统计和用户系统。
3. 更新题库需要重新发版。
4. 不适合复杂推荐算法。

这和当前 MVP 目标是匹配的。

## 5. 推荐技术栈

当前方案：

1. 前端框架：原生 HTML / CSS / JavaScript
2. 样式方案：普通 CSS
3. 路由方案：页面内视图切换
4. 状态管理：运行时内存状态
5. 部署：GitHub Pages

选择该方案的原因：

1. 无需安装依赖和构建步骤。
2. 直接打开 `index.html` 即可运行。
3. 部署到 GitHub Pages 更轻。
4. 更适合当前 MVP 的规模。

## 8. 核心模块设计

1. 话题读取模块
职责：
   - 维护首发题库
   - 维护分类配置

2. 随机选择模块
职责：
   - 从候选话题中随机选择
   - 按分类过滤候选话题
   - 在当前页面会话中尽量避免连续重复

3. 页面控制模块
职责：
   - 管理首页和分类页切换
   - 渲染话题卡片
   - 绑定换题和跳过交互

## 9. 关键逻辑建议

### 9.1 首页换题逻辑

1. 读取全部可用话题。
2. 排除当前会话内已跳过话题。
3. 从候选集中随机选一条。
4. 若与当前题目重复，则重新抽取。
5. 候选较少时允许回退为全量重抽。

### 9.2 分类页换题逻辑

1. 根据当前分类过滤候选题。
2. 排除当前会话内已跳过话题。
3. 从该分类候选集中随机选一条。
4. 若与当前题目重复，则重新抽取。
5. 候选过少时允许重复出现。

## 10. 测试策略

当前测试放在 `test` 目录下，覆盖：

1. 每个分类都有可用话题。
2. 跳过话题会从候选池中过滤。
3. 某分类全部跳过后会回退，避免页面无题可出。
4. 随机选择在可行时避免连续重复。
5. 首页引用的 CSS 和 JS 资源真实存在。
6. 首页脚本按数据、引擎、应用的顺序加载。
7. 页面没有使用模块脚本，支持直接打开本地 HTML。

## 11. GitHub 相关方案

MVP 阶段 GitHub 承担这几件事：

1. 代码仓库托管
2. 题库数据存储
3. GitHub Pages 静态部署

建议流程：

1. 本地开发
2. 提交到 GitHub 仓库
3. 打开 GitHub Pages
4. 选择默认分支根目录部署
5. 手机浏览器直接访问公开地址

## 12. 后续扩展预留

虽然 MVP 不做后端，但结构上保留升级空间。

后续可扩展方向：

1. 接入 Supabase 或 Firebase 做账号和同步
2. 把题库迁移到远程数据库
3. 加入服务端推荐逻辑
4. 加入行为埋点和统计分析
5. 增加后台内容管理能力

所以当前代码里已经把“题库读取”“随机引擎”和“页面展示”分层，没有把所有逻辑直接写死在页面组件中。

## 13. 技术结论

当前最合适的 MVP 技术路线是：

原生 HTML / CSS / JavaScript + 本地静态题库 + GitHub Pages

这是一个足够轻、足够快、足够适合手机访问，也最符合当前 MVP 范围的方案。
